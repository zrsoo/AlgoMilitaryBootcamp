# LC <11> — <Container With Most Water>
- **Pattern:** <Two Pointers — converging from both ends>
- **Brute force:** <Try every pair `(i, j)`, compute `min(h[i], h[j]) * (j - i)`, track max> — O(n²) time, O(1) space
- **Optimized:** <Two pointers `l=0`, `r=n-1`. At each step, compute area with current pair, then move the pointer at the **shorter** line inward (move the taller one and width shrinks while height ceiling can't grow).> — O(n) time, O(1) space
- **Key insight:** <The area is bounded by the **shorter** of the two walls times the width. Moving the taller pointer inward can never improve the area: width strictly decreases and the height ceiling (the shorter wall) cannot increase. Moving the shorter pointer is the only move that *might* yield a better area — so we safely discard all pairs involving the current shorter wall as the limiting factor.>
- **Edge cases I had to handle:** <(1) n == 2 → loop runs once, returns the single pair's area; (2) equal heights at `l` and `r` → either pointer can move (code picks `r--` via the `else` branch — both are correct, but pick one and be consistent); (3) all equal heights → answer is `h[0] * (n-1)`; (4) monotonic (sorted) heights → optimal pair is `(0, n-1)` for ascending; algorithm still finds it via pointer convergence; (5) initialized `maxCapacity = -1` works since areas are non-negative, but `0` is more idiomatic.>
- **Where I got stuck and for how long:** <Needed the editorial. Knew it was two-pointer but couldn't justify *which* pointer to move and *why* moving the taller one was safe to skip. The "shorter wall is the bottleneck, taller wall provides no upside when moved" argument is the whole problem — without it, you can't prove the greedy is optimal and end up second-guessing back to O(n²).>
